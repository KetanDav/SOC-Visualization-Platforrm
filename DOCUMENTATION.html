Introduction & Overview

What is the SOC Visualization Platform?

The SOC Visualization Platform is a full-stack, AI-assisted security operations centre dashboard designed for enterprise environments. It ingests telemetry data from multiple network vendors, normalises it into a unified schema, and presents it through rich interactive visualisations — all enriched by on-demand AI incident analysis.

📡

Multi-Vendor Ingestion

Parses CSVs from QRadar, Cisco SNA, Arista, Cisco ISE, Cisco DNAC, and Cisco APIC.

🧠

AI-Powered Analysis

Ask the AI anything about your telemetry, or trigger automated incident analysis per IP entity.

📊

Rich Visualisations

Communication graphs, Sankey charts, timeline sparklines, and IP activity charts via ECharts & Cytoscape.

🔁

Hot Reload Config

AI provider and model config reloads from .env on every request — no server restart needed.

📤

Upload & Export

Drag-and-drop CSV upload per vendor, normalise in memory, and export as a ZIP of CSVs.

🏢

Workspaces

3 npm workspaces — frontend, backend, and shared — share TypeScript types via @soc/telemetry-shared.

ℹ️
Port Layout

The Vite dev server runs on port 5000 and proxies all /api/* requests to http://127.0.0.1:3001 where the Express backend listens. In production, Express serves the built frontend bundle directly.

Install, configure, and run the platform
  1. Clone & install all dependencies
    terminalbash

Hot .env Reload

analysisConfig.ts exports a reloadEnvIfNeeded() function that is called at the start of every analysis/query request. It re-reads .env from disk on each invocation, so you can change the AI provider, model, temperature, or any other variable at runtime without restarting the backend server.

  3. Ensure Ollama is running: ollama serve
  4. Set ANALYSIS_PROVIDER=ollama in .env (this is the default)

Gemini Model Fallback Chain

geminiClient.ts attempts models in sequence, falling back to the next on rate-limit or model-not-found errors:

geminiClient.ts — fallback orderTypeScript
const FALLBACK_CHAIN = [
  "gemini-flash-lite-latest",         // ← first attempt
  "gemini-3.1-flash-lite",
  "gemini-3.1-flash-lite-preview",
  "gemini-flash-latest",
  process.env.GEMINI_MODEL,            // ← GEMINI_MODEL env var (last resort)
];

If all models in the chain fail, the error is surfaced to the client with a descriptive message.

⚠️
Ollama Timeout

The default OLLAMA_TIMEOUT_MS=120000 (2 minutes) is intentionally generous. Local models — especially on CPU — can take 30–90 seconds for the first token of a long context request. If you experience consistent timeouts, try a smaller model (e.g., qwen3:1.7b) or increase the timeout further.

SI Number Parsing

normalize.ts understands SI-suffixed numbers in raw CSV columns. For example, "148.17 K" is parsed as 148,170 and "2.3 M" as 2,300,000. This handles QRadar and SNA exports that abbreviate large byte counts.

REST API Endpoints

All Express routes exposed on port 3001
GET /api/health Server liveness check

Returns {"status":"ok","uptime":<seconds>}. Use to verify the backend is reachable before rendering the UI.

GET /api/payload Full normalised dataset

Returns the complete NormalizedData object: all assets, flows, events, and alerts ingested from the data/ directory (+ any uploaded files). This is the primary data source for the frontend.

POST /api/analysis/incident AI incident analysis report

Body: {"context": IncidentAnalysisContextPackage, "provider": "gemini"|"ollama"}

Assembles a structured prompt from the context package, calls the configured AI provider, strips code fences, and parses the JSON response into a validated incident report. Returns a RATReport object.

POST /api/query Free-form AI question (Ask AI tab)

Body: {"question": string, "context": NormalizedData, "provider": "gemini"|"ollama"}

Injects the full telemetry dataset as context, applies the SOC analyst system prompt, and returns the AI's markdown-formatted answer as a plain string.

POST /api/upload Upload vendor CSVs

Body: JSON object where keys are vendor names (qradar, sna, …) and values are base64-encoded CSV content strings. The server decodes, saves to data/uploads/, and triggers normalisation.

GET /api/upload/status Upload processing status

Returns the current upload state: {"status":"idle"|"processing"|"done"|"error", "message": string}. The frontend polls this after POSTing to /api/upload.

DELETE /api/upload Clear uploaded files

Deletes all files from data/uploads/ and resets the normalised data to the baseline data/ CSVs only. Returns {"cleared": true}.

GET /api/raw Raw vendor rows for an IP

Query params: ?vendor=qradar&ip=10.10.10.21

Returns the original, un-normalised CSV rows from the specified vendor that contain the given IP address. Used by the Raw Logs tab to show source-of-truth data.

GET /api/export/csv Download normalised data as ZIP

Streams a ZIP archive (Content-Disposition: attachment; filename=soc_export.zip) containing four CSV files: assets.csv, flows.csv, events.csv, alerts.csv. Built with archiver via zipBuilder.ts.

Data Normalization Engine

How raw CSVs become NormalizedData

normalize.ts is the core ETL engine. It processes all CSV files from data/ and data/uploads/, applies vendor-specific parsers, and produces a unified NormalizedData object.

Processing Pipeline

  1. Load CSVs — loadCsv.ts reads each file, strips BOM, and returns an array of row objects keyed by column name.
  2. Vendor dispatch — Each file is routed to a vendor-specific parser function based on the filename prefix (qradar_, sna_, etc.).
  3. Field extraction — Parsers map vendor column names to normalised field names. Missing optional fields default to undefined.
  4. SI number parsing — Byte/packet counts like "148.17 K" are parsed to integers. Supported suffixes: K (×1,000), M (×1,000,000), G (×1,000,000,000).
  5. Asset merging — All assets are deduplicated by ip. When the same IP appears in multiple vendors, metadata is merged and communicationVolume is summed.
  6. Risk scoring — Each asset receives a computed risk score based on alert count, maximum alert severity, and communication volume percentile.
normalize.ts — SI parser exampleTypeScript
function parseSI(val: string): number {
  const m = val.trim().match(/^([\d.]+)\s*([KMG])?$/i);
  if (!m) return 0;
  const n = parseFloat(m[1]);
  const mul = { K: 1e3, M: 1e6, G: 1e9 }[m[2]?.toUpperCase()] ?? 1;
  return Math.round(n * mul);
}

AI Analysis Engine

Incident analysis and Ask AI internals

IncidentAnalysisContextPackage

When the user clicks Run Analysis on an entity, incidentAnalysis.ts assembles a IncidentAnalysisContextPackage containing:

targetIp relatedAssets relatedFlows relatedEvents relatedAlerts totalAssets totalFlows timestamp

Only records relevant to the target IP are included to keep the prompt size manageable.

Prompt Engineering (incidentPrompt.ts)

The system prompt instructs the model to return a strict JSON object — no prose wrapping. Key response fields include riskLevel, summary, indicators, recommendations, and confidence. The user prompt serializes the context package as JSON.

JSON Parsing & Validation

The raw AI response is processed by a stripCodeFences() helper that removes markdown backtick blocks before JSON.parse(). Field validation ensures required keys exist; missing optional fields are defaulted. If parsing fails entirely, a structured error response is returned to the frontend.

Frontend Architecture

App.tsx responsibilities and data flow

App.tsx — Root Component

App.tsx is the single source of truth for all global state. Its key responsibilities:

  • Data fetching: Calls GET /api/payload on mount and stores the full NormalizedData in a useState hook.
  • Filter state: Maintains selectedIp, severityFilter, vendorFilter, searchQuery, and activeTab.
  • Derived data (computed client-side):
    • filteredFlows — flows involving selectedIp or matching searchQuery
    • filteredEvents — events filtered by severityFilter and vendorFilter
    • filteredAlerts — alerts filtered by severityFilter
  • Tab routing: Renders the correct tab component based on activeTab — no React Router needed.
  • Provider forwarding: Passes the current AI provider selection down to AskAITab and IncidentAnalysisPanel.
ℹ️
No External State Library

The platform intentionally uses only React's built-in useState and useMemo. All derived state (filteredFlows, etc.) is computed with useMemo to avoid unnecessary recalculations. This keeps the bundle small and the data flow easy to trace.

Dashboard Tabs Usage Guide

How to use each tab effectively

📈 Dashboard Tab

Start here for a high-level health overview. The KPI cards give instant situational awareness. Use the protocol distribution chart to spot unusual traffic (e.g., unexpected ICMP spikes). The top-10 flows table shows your biggest bandwidth consumers.

🕸️ Communication Graph

Use this to identify unusual communication patterns. Nodes sized by communication volume; coloured by max alert severity (green → amber → red → crimson). Zoom and pan freely. Click a node to open the Entity Workspace for that IP. Use the filter bar to isolate high-severity IPs.

🔍 IP Activity

Deep-dive into a specific IP. The timeline shows inbound vs. outbound byte rates — sudden spikes are worth investigating. The port heatmap reveals which destination ports are being hit. Switch to this tab after clicking a suspicious node in the graph.

📋 Raw Logs

Use this to verify normalisation fidelity. Select a vendor and search for an IP to see the exact, un-transformed rows. Useful for validating that numeric conversions (e.g., SI parsing) produced the expected results.

💬 Ask AI

Ask natural-language questions about your telemetry. Example questions: "Which IPs have the highest outbound bytes?", "Are there any lateral movement indicators?", "Summarise all critical alerts." The full dataset is injected automatically — you don't need to copy/paste any data.

🧑‍💼 Entity Workspace — AI Analysis Sub-Tab

For any selected IP, click Run Incident Analysis to generate a structured RAT (Risk / Action / Threat) Report. The report includes a risk level badge, a plain-English summary, a list of threat indicators (with IPs highlighted), actionable recommendations, and a confidence percentage. This is the platform's primary threat-hunting workflow.

Troubleshooting

Common issues and how to resolve them

❓ No data appears on the Dashboard

Check that data/ contains at least one CSV file with the correct vendor prefix. Open the browser DevTools Network tab and inspect the GET /api/payload response. If it returns empty arrays, verify that normalize.ts can read the file paths (try running npm run dev from the repository root, not a subdirectory).

❓ Ask AI returns raw HTML or an error instead of an answer

This usually means Vite's proxy is not reaching the backend. Ensure npm run dev started both the backend and frontend. Check that Express is actually running on port 3001 by visiting http://localhost:3001/api/health directly. If the backend is down, the proxy returns Vite's own HTML error page.

❓ Missing GEMINI_API_KEY error

You have ANALYSIS_PROVIDER=gemini set but no GEMINI_API_KEY in your .env. Either add the key or switch to ANALYSIS_PROVIDER=ollama. Remember: analysisConfig.ts hot-reloads on every request, so you can add the key to .env without restarting the server.

❓ Ollama requests time out

Local model inference is slow, especially on CPU-only machines. Try: (1) increasing OLLAMA_TIMEOUT_MS in .env, (2) switching to a smaller model (ollama pull qwen3:1.7b), or (3) ensuring Ollama is already warmed up by running a quick ollama run qwen3:4b "hello" first.

❓ Upload not working

Ensure the data/uploads/ directory exists and is writable by the Node.js process. On Windows, check that no antivirus is blocking file creation in that directory. Also verify the POST body is valid JSON with base64 values — the frontend's UploadModal does this encoding automatically.

❓ Port conflict (EADDRINUSE)

Another process is using port 3001 or 5000. To find and kill it on Windows: netstat -ano | findstr :3001 then taskkill /PID <pid> /F. Alternatively, change PORT=3001 in .env and update vite.config.ts's proxy target to match.

❓ TypeScript compilation errors

If you see TS errors referencing @soc/telemetry-shared, run npm install from the repo root to ensure workspace symlinks are in place. Then run npm run build in the shared workspace first, as frontend and backend depend on its compiled output.
